# 실습 시나리오 (교육용)

각 실습: [목표 / 절차 / 예상 결과 / 실패 사례]. 반드시 **테스트 계정**으로 진행하고, 운영 데이터를 바꾸지 않는다.

---

## 실습 1 — 청소관리·하자접수만 가능한 시설 담당자
- 목표: 선택(restrictive) 방식으로 최소 메뉴 권한 만들기.
- 절차:
  1. 권한관리 > 신규 권한, 이름 "시설 하자 담당", 적용 방식 = **선택한 메뉴만 허용**
  2. 저장 후 수정에서 **운영관리 > 청소관리, 하자접수**만 메뉴·기능 선택
  3. 테스트 계정에 배정 → 재로그인
- 예상 결과: 사이드바에 운영관리(청소관리·하자접수)만 표시. 다른 메뉴 URL 접근 시 리다이렉트/차단.
- 실패 사례: 메뉴를 선택하지 않아 아무 메뉴도 안 보임 / 방식을 "추가"로 해서 기본 역할 메뉴까지 보임.

## 실습 2 — 기숙사관리 기본 + 시험 조회 기능 추가
- 목표: 추가(additive) 방식으로 조회 기능만 얹기.
- 절차:
  1. 신규 권한 "시험 조회 추가", 적용 방식 = **기존 권한에 추가**
  2. 수정에서 **시험관리 > 시험 응시관리**의 조회·상세만 선택(등록·수정·승인 제외)
  3. 기숙사 담당 계정에 배정 → 재로그인
- 예상 결과: 기존 기숙사 메뉴 유지 + 시험 응시관리 조회 가능. 등록/수정 버튼 없음.
- 실패 사례: 쓰기 기능까지 선택해 조회 전용 의도가 깨짐.

## 실습 3 — 평택 남성 기숙사만 조회 가능한 사용자
- 목표: 데이터 범위(지역+성별) 교집합 설정.
- 절차:
  1. 신규 권한 "평택 남성 조회", 적용 방식 = 선택, 메뉴 = 기숙사관리 > 입주자(조회·상세)
  2. 데이터 범위: 지역 **평택**, 성별 **남성**
  3. 테스트 계정 배정 → 재로그인 → 입주자 목록·Excel·대시보드 확인
- 예상 결과: 평택 **이면서** 남성 입주자만 목록·Excel·대시보드에 표시.
- 실패 사례: 범위 미설정으로 0건 / "전체" 선택으로 조건이 사라짐.

## 실습 4 — PM 공정만 처리 가능한 시험 담당자
- 목표: 시험 공정 범위 설정(서버 강제).
- 절차:
  1. 신규 권한 "PM 공정 담당", 메뉴 = 시험관리 > 시험 응시관리(조회·등록·수정), 데이터 범위 = 공정 **PM**
  2. 테스트 계정 배정 → 재로그인
- 예상 결과: PM 공정 데이터만 조회·수정. DM 공정 상세 URL·수정 차단(서버 거부).
- 실패 사례: 공정 미지정 → 조회 0건.

## 실습 5 — 퇴사자 권한 회수 및 계정 비활성화
- 목표: 퇴사 처리 절차 숙지.
- 절차:
  1. 대상(테스트) 계정 수정 → 추가 권한 **모두 선택 해제** 후 저장
  2. 계정 **사용중지(비활성화)**
  3. 재로그인 시도로 차단 확인, 감사로그 확인
- 예상 결과: 로그인 차단, 추가 권한 회수, 감사로그에 변경 기록.
- 실패 사례: 마지막 관리자 계정을 비활성화하려다 시스템 차단(정상) — 다른 관리자 선지정 필요.

## 실습 6 — 권한 없는 데이터 접근 기록 확인
- 목표: 감사로그·서버 로그로 접근 시도 확인.
- 절차:
  1. 범위 제한 테스트 계정으로 범위 밖 데이터/메뉴 접근 시도
  2. 관리자 계정으로 감사로그(custom_role_audit_logs / security_audit_logs) + Supabase 로그(정상 403/거부) 확인
- 예상 결과: 클라이언트 차단 + 서버 거부(시험/RLS 적용 영역) 확인.
- 실패 사례: 감사로그가 아닌 곳에서 찾아 못 찾음 → 위치 재확인.
