# 권한 사고 대응 · 장애 확인 순서

대상: 보안 담당자, 관리자

---

## 1. 사고별 대응 매뉴얼

각 사고는 [심각도 / 즉시 조치 / 확인 로그 / 임시 차단 / 원인 확인 / 복구 / 재발 방지 / 보고 대상] 순.

### 1) 권한 없는 메뉴가 보이는 경우
- 심각도: **High**
- 즉시 조치: 해당 사용자 계정 사용중지 또는 추가 권한 회수
- 확인 로그: 감사로그(최근 권한 변경), 해당 계정의 배정 권한
- 임시 차단: 문제 Custom Role 사용중지
- 원인 확인: 적용 방식(추가/선택) 오설정, 잘못된 메뉴 권한 부여
- 복구: 올바른 권한으로 재설정 후 재로그인 확인
- 재발 방지: 신규/수정 권한은 테스트 계정 검증 후 배포
- 보고: 시스템 관리자, 보안 담당

### 2) URL 직접 접근이 가능한 경우
- 심각도: **High**
- 즉시 조치: 해당 사용자 권한 회수, 사고 재현 기록
- 확인 로그: 접근 대상 메뉴, 배정 권한, Network 403 여부
- 원인 확인: 메뉴 권한(선택 메뉴) 과다, 라우트 접근 예외
- 복구: 권한 정정, 재로그인 후 URL 접근 차단 확인
- 재발 방지: 역할별 접근 매트릭스 재검토
- 보고: 시스템 관리자

### 3) 범위 밖 데이터가 보이는 경우
- 심각도: **High** (기숙사 서버 강제 미적용 시 Critical 가능)
- 즉시 조치: 해당 계정 사용중지, 데이터 범위 재확인
- 확인 로그: custom_role_scopes 설정, 사용자 최종 범위, 다운로드 이력
- 임시 차단: 문제 Custom Role 사용중지 또는 범위 축소
- 원인 확인: 데이터 범위 미설정/과다, **서버 범위 강제(RLS) 적용 여부(개발팀 확인)**
- 복구: 범위 정정, 서버 강제 적용 검토
- 재발 방지: 민감 기숙사 범위는 서버 강제 적용 확인 후 부여
- 보고: 보안 담당, 개인정보 책임자

### 4) Excel에 전체 데이터가 포함된 경우
- 심각도: **Critical**
- 즉시 조치: 다운로드 권한 회수, 유출 파일 회수·폐기 지시
- 확인 로그: 다운로드 이력, 해당 계정 범위 설정
- 원인 확인: 데이터 범위 미적용, 다운로드 함수 우회
- 복구: 범위 재설정 후 다운로드 재검증(화면 개수 = 파일 개수)
- 재발 방지: 다운로드 권한 별도 관리, 범위 검증
- 보고: 보안·개인정보 책임자, 경영진

### 5) admin이 아닌 사용자가 권한관리에 접근한 경우
- 심각도: **Critical**
- 즉시 조치: 해당 계정 사용중지
- 확인 로그: 계정 역할, 배정 권한(권한관리 부여 여부)
- 원인 확인: 관리자 전용 메뉴 노출/부여
- 복구: 권한 회수, 관리자 전용 접근 재확인
- 보고: 시스템 관리자, 보안

### 6) 퇴사자 계정이 활성 상태인 경우
- 심각도: **High**
- 즉시 조치: 계정 사용중지, 추가 권한 회수
- 확인 로그: 최근 로그인 이력, 감사로그
- 재발 방지: 퇴사 처리 체크리스트·월간 점검 강화
- 보고: 인사·보안

### 7) 역할 변경 후 이전 권한이 남은 경우
- 심각도: **Medium**
- 즉시 조치: 대상 사용자 재로그인/새로고침 안내
- 확인 로그: 배정 권한 현재 상태
- 원인 확인: 권한 변경은 다음 로그인/새로고침 후 적용(정상 지연) 또는 회수 누락
- 복구: 재로그인 후에도 남으면 회수 누락 → 재설정
- 보고: 시스템 관리자

### 8) 사용자 A 권한이 사용자 B에게 보이는 경우
- 심각도: **Critical**
- 즉시 조치: 두 계정 로그아웃 유도, 사고 재현 기록
- 확인 로그: 각 계정 배정 권한, 브라우저/세션
- 원인 확인: 세션 혼선(정상 구현상 로그아웃 시 캐시 리셋됨 — 재현되면 즉시 보고)
- 복구: 재로그인 후 정상 여부 확인
- 보고: 보안, 개발팀

## 2. 장애 발생 시 확인 순서 (15단계)
1. 사용자 이메일/아이디 확인
2. 기본 역할 확인
3. 배정된 Custom Role 확인(사용자관리 > 추가 권한)
4. 활성·삭제 상태 확인
5. 적용 방식(추가/선택) 확인
6. 메뉴 권한 확인
7. 기능 권한 확인
8. 데이터 범위 확인
9. **최종 권한 미리보기** 확인
10. 로그아웃 후 재로그인
11. 브라우저 캐시/새로고침
12. Network 401/403 확인(정상 권한 차단 vs 오류 구분)
13. 감사로그 확인(최근 변경)
14. Supabase RLS/RPC 확인(개발팀/서버 로그)
15. 최근 권한 변경 이력 확인(누가·언제)

---

## 3. 대응 원칙
- **Critical**: 신규 트래픽 영향 확인 → 이전 배포 Rollback 검토 → 원인 파악 → DB 영향 확인 → **임시 권한 우회 금지** → 수정 후 재검수.
- **High**: 해당 기능/권한 제한 → 최소 수정 → 동일 유형 재테스트 → 재배포 전 핵심 검수 재실행.
- **Medium/Low**: 운영 영향 판단 → 수정 목록 작성 → 우선순위 지정.
> 원인 확인 전 **반복 재배포 금지**. **권한 없는 경우 fail-open(무조건 허용) 금지** — 실패 시 차단(fail-closed).
