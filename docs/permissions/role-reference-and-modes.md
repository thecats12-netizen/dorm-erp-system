## 1. 역할별 기준표

### 관리자 (시스템 값: admin)
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 시스템 운영 책임자, 총무 책임자 |
| 기본 목적 | 전체 운영·권한 관리 |
| 접근 메뉴 | 전체 |
| 가능 기능 | 조회·등록·수정·삭제·승인·반려·전체 다운로드·권한관리 |
| 다운로드 | 가능(전체) |
| 데이터 범위 | 제한 없음 |
| 부여 승인자 | 최고관리자 |
| 부여 기간 | 상시(최소 인원) |
| 정기 검토 | 월 1회 |
| 주의사항 | 최소 인원 유지, 마지막 관리자 강등·비활성화 불가 |

### 뷰어 (시스템 값: viewer)
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 조회만 필요한 임원·유관부서 |
| 기본 목적 | 데이터 열람 |
| 접근 메뉴 | 시스템(관리) 그룹 제외 대부분 |
| 가능 기능 | 조회·상세 (등록·수정·삭제 불가) |
| 다운로드 | 기본 불가(필요 시 추가 권한으로 별도 부여) |
| 데이터 범위 | 제한 없음(조회) |
| 부여 승인자 | 부서 관리자 |
| 정기 검토 | 분기 1회 |
| 주의사항 | 쓰기 기능이 필요하면 뷰어가 아니라 별도 권한 검토 |

### 기숙사 관리자 (시스템 값: dorm_manager)
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 담당 기숙사 운영자 |
| 기본 목적 | 담당 기숙사 청소·하자 처리 |
| 접근 메뉴 | 운영관리 > 청소관리·하자접수(담당 기숙사) |
| 가능 기능 | 기존 정책 유지 |
| 다운로드 | 기존 정책 유지 |
| 데이터 범위 | 담당 기숙사 |
| 부여 승인자 | 관리자 |
| 정기 검토 | 분기 1회 |
| 주의사항 | 담당 기숙사 지정 필요 |

### 하자접수 담당자 (시스템 값: maintenance_reporter)
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 시설·하자 신고 담당 |
| 기본 목적 | 청소·하자 접수 처리 |
| **접근 메뉴** | **운영관리 > 청소관리, 하자접수 (그 외 원칙적으로 불가)** |
| 가능 기능 | 기존 정책 유지 |
| 다운로드 | 기존 정책 유지(이미지 등) |
| 데이터 범위 | 기존 담당 기숙사·지역 |
| 부여 승인자 | 관리자 |
| 정기 검토 | 분기 1회 |
| 주의사항 | **추가 권한 부여 불가(시스템 차단). 다른 메뉴 노출 시 즉시 사고 대응** |

### 추가(additive) Custom Role 사용자
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 기본 업무 + 일부 추가 업무 담당 |
| 최종 권한 | 기본 역할 권한 **+** 선택 권한(합집합) |
| 주의사항 | 기본 역할 권한이 유지되므로 **권한이 넓어질 수 있음** |

### 선택(restrictive) Custom Role 사용자
| 항목 | 내용 |
|---|---|
| 주요 사용자 | 외부 협력업체, 단기 계약직, 특정 업무 전담자 |
| 최종 권한 | 선택한 메뉴·기능만(기본 역할 권한 제외) |
| 주의사항 | **필수 메뉴 누락 시 업무 불가**. 데이터 범위 없으면 해당 메뉴 0건 |

### 여러 Custom Role 사용자
| 항목 | 내용 |
|---|---|
| 메뉴·기능 | 선택(restrictive)이 하나라도 있으면 **선택 우선**(그 권한들만). 모두 추가(additive)면 합집합 |
| 데이터 범위 | 같은 유형 합집합, 다른 유형 교집합 |
| 주의사항 | 조합이 복잡하면 **최종 권한 미리보기**로 반드시 확인 |

## 2. 적용 방식 가이드 — 추가(additive) vs 선택(restrictive)

### 기존 권한에 추가 (시스템 값: additive)
- **뜻**: 기본 역할을 유지하면서 선택한 메뉴·기능·범위를 **더한다**.
- **적합 예**: 기숙사 담당자에게 **시험 조회 권한만** 추가 / 일반 관리자에게 **특정 보고서 다운로드만** 추가.
- **주의**: 기본 역할 권한까지 유지되어 **권한이 넓어질 수 있음**.

### 선택한 메뉴만 허용 (시스템 값: restrictive)
- **뜻**: 선택한 메뉴·기능만 사용, 나머지는 숨김(기본 역할 권한 제외).
- **적합 예**: **외부 협력업체, 시설 신고 담당자, 단기 계약직, 특정 업무 전담자**.
- **주의**: **필수 메뉴를 빠뜨리면 업무가 불가능**. 데이터 범위를 지정하지 않으면 해당 메뉴가 0건으로 보일 수 있음.

### 어떤 방식을 고를까? (판단표)
| 상황 | 권장 방식 |
|---|---|
| 기존 업무에 소량 추가 | 기존 권한에 추가(additive) |
| 특정 업무만, 나머지는 숨겨야 함 | 선택한 메뉴만 허용(restrictive) |
| 외부·단기·협력업체 | 선택한 메뉴만 허용(restrictive) |
| 임원/유관부서 조회 | 뷰어 역할 또는 additive 조회 |
| 넓은 접근이 필요한 내부 담당 | 기존 권한에 추가(additive) |

## 3. 데이터 범위 설정 가이드

### 기숙사관리 범위
- 전체 데이터 / 특정 지역 / 특정 성별 / 특정 기숙사 / 특정 담당자 / 본인이 등록한 데이터

### 시험관리 범위
- 전체 데이터 / 특정 시험 공정 / 특정 담당자 / 본인이 등록한 데이터

### 결합 규칙
- **같은 유형 여러 값 = 합집합**
- **다른 유형 조합 = 교집합**

예시:
- `서울 + 경기` → 서울 **또는** 경기
- `서울 + 여성` → 서울 **이면서** 여성
- `A기숙사 + B기숙사 + 담당자 홍길동` → (A **또는** B) **이면서** 담당자 홍길동

### 잘못된 범위 설정 사례
- ❌ **범위를 하나도 지정하지 않고 선택(restrictive) 권한 부여** → 해당 메뉴 데이터가 0건(업무 불가).
- ❌ **전체 데이터 + 특정 지역을 동시에 의도** → "전체"를 선택하면 다른 조건은 자동 해제된다(전체가 우선).
- ❌ **너무 넓은 범위(전체 지역·전체 기숙사)를 협력업체에 부여** → 최소 권한 원칙 위반.
- ❌ **담당자/본인 등록 범위를, 담당자 정보가 없는 데이터에 적용** → 조회 0건 가능. 실제 담당 컬럼 존재 여부 확인.
