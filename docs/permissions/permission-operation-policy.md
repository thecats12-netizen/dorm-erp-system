# 권한 운영 정책서

대상: 시스템 최고관리자, 일반 관리자, 보안·감사 담당자
목적: 권한 부여·회수·검토의 공통 기준을 정한다. 모든 관리자는 이 정책을 준수한다.

---

## 1. 권한 운영 10대 기본 원칙

1. **최소 권한 원칙** — 업무에 필요한 최소 메뉴·기능·데이터 범위만 부여한다.
2. **업무 분리 원칙** — 승인자와 처리자를 가능하면 분리한다(예: 권한 신청자 ≠ 최종 승인자).
3. **퇴사 즉시 회수** — 퇴사일/계약 종료 시점에 계정 비활성화 + 추가 권한 회수.
4. **임시 권한 만료 지정** — 단기·협력업체 권한은 **적용 시작일/종료일** 을 지정한다.
5. **관리자 권한 제한** — **관리자(admin)** 는 최소 인원만 보유(권장 2~3명). 마지막 관리자 계정은 시스템이 비활성화/강등을 차단한다.
6. **다운로드 권한 별도 관리** — Excel/CSV/PDF/인쇄 권한은 일반 조회와 구분해 부여한다.
7. **개인정보 접근 제한** — 입주자·시험 응시자·담당자 개인정보 접근은 최소화한다.
8. **데이터 범위 구체화** — 지역·기숙사·공정·담당자 범위를 가능한 한 구체적으로 설정한다.
9. **정기 재검토** — 월 1회 권한 점검, 분기 1회 보안 점검을 수행한다.
10. **변경 이력 확인** — 권한 변경 전/후와 변경자는 감사로그로 확인한다.

---

## 2. 권한 구조 개요 (비개발자용)

- **기본 역할(시스템 역할)**: 시스템에 고정된 4종. 관리자가 이름·권한을 바꿀 수 없고 **복제만** 가능하다.
  - 관리자(시스템 값: admin), 뷰어(viewer), 기숙사 관리자(dorm_manager), 하자접수 담당자(maintenance_reporter)
- **사용자 정의 권한(Custom Role)**: 업무별로 관리자가 직접 만드는 권한. 메뉴·기능·데이터 범위를 지정한다.
- **적용 방식**:
  - **기존 권한에 추가(additive)** — 기본 역할을 유지하면서 선택한 권한을 더한다.
  - **선택한 메뉴만 허용(restrictive)** — 선택한 메뉴·기능만 사용하고 나머지는 숨긴다(기본 역할 권한은 최종 계산에서 제외).

---

## 3. 역할별 운영 정책 요약

| 역할 | 부여 승인자 | 부여 기간 | 정기 검토 | 핵심 주의 |
|---|---|---|---|---|
| 관리자(admin) | 최고관리자 | 상시(최소 인원) | 월 1회 | 마지막 관리자 강등·비활성화 불가 |
| 뷰어(viewer) | 부서 관리자 | 상시 | 분기 1회 | 조회 전용 유지 |
| 기숙사 관리자(dorm_manager) | 관리자 | 상시 | 분기 1회 | 담당 기숙사 기준 유지 |
| 하자접수 담당자(maintenance_reporter) | 관리자 | 상시 | 분기 1회 | **청소관리·하자접수만**(아래 4항) |
| 추가(additive) Custom Role | 부서 관리자→관리자 | 업무 기간 | 월 1회 | 권한이 넓어질 수 있음 |
| 선택(restrictive) Custom Role | 관리자 | 계약 기간 | 월 1회 | 필수 메뉴 누락 주의 |

역할별 상세 기준표(메뉴·기능·다운로드·데이터 범위)는 [role-reference-and-modes.md](role-reference-and-modes.md) 참고.

---

## 4. 하자접수 담당자(maintenance_reporter) 보호 정책 【필독】

- 하자접수 담당자 계정은 **항상 아래 메뉴만** 보인다.
  - **운영관리 > 청소관리**
  - **운영관리 > 하자접수**
- 그 외 메뉴(기본정보·기숙사관리·자산관리·시험관리·군대관리·시스템)는 **원칙적으로 접근 불가**.
- 이 계정에는 **사용자 정의 권한(추가 권한)을 부여하지 않는다.** 시스템이 부여 자체를 차단한다.
- 등록/수정/삭제 등 기존 기능 정책을 임의로 바꾸지 않는다.
- 데이터 범위 설정의 영향도 받지 않는다(기존 담당 기숙사·지역 유지).

> ⚠️ 하자접수 담당자에게 다른 메뉴가 보이거나 URL로 접근되면 **즉시 사고 대응**([permission-incident-response.md](permission-incident-response.md)).

## 5. 데이터 범위·다운로드 정책

- **데이터 범위**: 사용자 정의 권한에 지역/성별/기숙사/시험 공정/본인 등록 데이터 범위를 지정할 수 있다.
  - 같은 유형 여러 값 = **합집합**(예: 서울, 경기 → 둘 다 허용)
  - 다른 유형 조합 = **교집합**(예: 서울 + 여성 → 서울이면서 여성만)
- **시험 공정 범위**는 서버에서 강제된다(우회 불가).
- **기숙사 지역/성별/기숙사 범위**는 현재 화면·기능 계층에서 적용된다. 외부에 신뢰하기 어려운 사용자에게 민감한 기숙사 범위를 부여할 경우, 서버 강제(RLS) 적용 여부를 운영자가 확인한다(개발팀 확인 항목).
- 다운로드 결과에는 **화면에 보이는 범위만** 포함된다. 전체 데이터가 포함되면 사고로 처리한다.
